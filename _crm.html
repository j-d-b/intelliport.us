CRM

Container Relocation Module

Minimizing Unproductive Container Moves

Background

Due to limited space within the storage yard, containers are stacked in columns that are only accessible from the top. Thus, in order to retrieve a container that is not on the top of a stack, the blocking containers above this container must be relocated to a different column.

The column selected for relocation of the blocking container can have a significant impact on the operational efficiency of the storage yard, as optimal placement can minimize the number of future relocations needed, which reduces wait time for customers, operating costs for port managers, and environmental impact.

Currently, the selection of a column for relocation of a blocking container is not optimized. Typically the blocking container is moved to a nearby column with the fewest containers.

Purpose

CRM automatically determines the optimal relocation move for blocking containers during the retrieval of a requested container, as well as the optimal placement within a block when stacking a new container.

It uses container retrieval order to maximize operational efficiency by minimizing the number of future relocations that are caused by the placement of blocking or new containers

Functionality

CRM has two functions:

  • Case 1. Determining placement within a block for the relocation of a blocking container during the retrieval of a requested container
  • Case 2. Determining placement within a block when stacking a new container

    • At a high level, CRM accepts in a container ID to retrieve (case 1) or a specified block to stack a new container (case 2) and outputs the optimized set of moves that should be taken to complete this action.

    Software Module

    CRM is a smart-stacking decision module that communicates with a Terminal Operating System (TOS).

    CRM receives data from the TOS about containers currently stacked in the yard to determine container departure order and sends the list of optimal moves to be performed to the TOS as instructions for use by crane operators.

    The module will continuously refresh its information and update the expected container retrieval order based on changes in the yard.

    If the terminal has implemented a Prediction System (PS) for container retrieval times or a Truck Appointment System (TAS), CRPM will also receive data from these systems.

    Components

    The module consists of four components:

    1. Departure Order Generator: Using the TOS data (and PS and TAS data, if available), generates an order of departure for all containers in the terminal. The data is updated in real-time, adapting to short-notice requests.
    2. Virtual Port: Stores data for all containers in the port, such as location (block, row, column, position) and retrieval order. Receives data from the TOS and Departure Order Generator. The Virtual Port is updated in real-time.
    3. Stacking Move Generator: Using the departure order from the Virtual Port and a placement block chosen by the planner, outputs the best column to stack a container.
    4. Retrieval Relocation Move Generator: Using the departure order data from the Virtual Port and the container ID to retrieve, generates a list of moves resulting in the retrieval of the desired container and outputs this to the TOS.

    Inputs

    CRM communicates in real-time with the TOS (and PS and TAS, if available) to maintain a store of current information about the terminal yard. This picture of the yard, stored in the Virtual Port, is supplemented with departure order information from the Departure Order Generator, and then forms the base informational input utilized by the optimization algorithm.

    For both CRM functional cases, retrieval and stack, specific user input is required in addition to the automatically processed yard information.

    • For the retrieval case, the planner will enter the container ID of the desired container to be retrieved.
    • For the new container stacking case, the operator must enter the block in which the new container will be stacked.

    To summarize, CRM requires the following inputs:

    • Container locations
    • Container departure order
    • Container ID (case 1) or Block ID (case 2)

    Outputs

    In both functional cases, CRM returns a list of operating instructions that will be sent to the TOS.

    For the stacking of a new container, the returned list will contain a single instruction.

    For container retrieval, the list will contain the instructions for any required relocations followed by the retrieval of the requested container.
